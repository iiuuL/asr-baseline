# 当前项目结构说明（供 API 改造规划使用）

## 1. 文档目的

这份说明的目标不是解释每一行代码，而是帮助另一个工程人员在**不直接阅读全部源码**的情况下，快速理解这个仓库现在是什么结构、核心逻辑在哪里、已经具备哪些“服务化基础”、还缺什么，进而据此制定“把当前项目整理为稳定 API 服务”的实施计划。

换句话说，这份文档回答的是下面几个问题：

- 这个仓库现在本质上是“脚本项目”还是“服务项目”
- 识别能力集中在哪些模块
- 当前在线接口做到什么程度
- 哪些目录是成熟能力，哪些目录只是预留或半成品
- 如果要继续推进 API 化，规划时必须考虑哪些现实约束

---

## 2. 项目一句话概述

这是一个基于 `FunASR + SenseVoiceSmall` 的语音识别项目，当前已经具备：

- 单文件识别能力
- 批量目录识别/////////能力
- 基础评测能力（生成预测 TSV、计算 WER/SER）
- 一个可运行但还比较初级的 `FastAPI` 在线服务原型

当前仓库的主体仍然是**本地脚本驱动的 ASR 推理/评测项目**，不是一个完整设计过的 API 产品。`service/` 目录表明项目已经开始往接口服务方向迁移，但迁移尚未完成。

---

## 3. 顶层目录结构与职责

当前仓库顶层大致如下：

```text
asr_baseline/
├─ asr_core/                  # 核心 ASR 推理与文本后处理逻辑
├─ service/                   # FastAPI 服务层（已能运行，但仍在迁移中）
├─ scripts/                   # 离线脚本：冒烟测试、批量识别、评测、数据清洗
├─ data/                      # 示例音频与评测样本
├─ logs/                      # 运行日志、批量错误日志
├─ outputs/                   # 评测输出目录（如 metrics.json）
├─ configs/                   # 预留配置目录，目前基本为空
├─ refs/                      # 预留参考资料目录，目前基本为空
├─ test/                      # 预留测试目录，目前基本为空
├─ requirements.txt           # pip 依赖
├─ environment.yml            # conda 环境定义
├─ run_service.cmd            # Windows 下启动 API 服务的脚本
├─ demo.html                  # 前端联调用的简单页面
├─ ASR接入.md                 # 面向接入方的说明文档
└─ API设计说明.md             # 目标 API 协议草案（与当前实现仍有差距）
```

从目录职责上看，可以把仓库分成 4 层：

1. `asr_core/`：真正的业务核心，负责模型加载、推理、标签解析、文本清洗。
2. `scripts/`：围绕核心能力做“离线调用”，用于测试、批处理、评测。
3. `service/`：围绕核心能力做“在线调用”，即 HTTP API。
4. `data/logs/outputs/`：围绕离线实验和联调的输入输出目录。

## 4. 核心模块说明

### 4.1 `asr_core/engine.py`

这是目前整个项目最核心的模块，负责把底层模型能力包装成统一的 Python 调用接口。

#### 主要职责

- 自动选择推理设备：优先 `cuda:0`，否则 `cpu`
- 在初始化阶段只加载一次模型，避免重复加载
- 调用 `FunASR` 的 `AutoModel.generate(...)`
- 统一抽取模型输出中的 `text` 和 `timestamp`
- 调用标签解析与文本清洗逻辑，生成结构化结果

#### 当前固定配置

- 模型：`iic/SenseVoiceSmall`
- VAD：`fsmn-vad`
- 默认语言：`zh`
- 默认 `use_itn=True`
- `batch_size_s=60`
- `merge_vad=True`
- `merge_length_s=15`

#### 对外暴露的关键方法

- `transcribe_file(audio_path, language="zh", use_itn=True)`
  - 输入：一个本地音频文件路径
  - 输出：结构化字典，包含：
    - `ok`
    - `raw_result`
    - `raw_text`
    - `text`
    - `text_clean`
    - `metadata`
    - `language`
    - `device`
    - `elapsed_ms`
    - `timestamp`
    - 失败时还有 `error`
- `get_status()`
  - 返回模型是否加载成功、当前设备、模型名等

#### 这个模块对 API 规划的重要性

API 层本质上只是这个模块的“传输适配层”。也就是说，后续要规划接口时，重点不是再发明新的识别逻辑，而是围绕 `InferenceEngine` 设计：

- 生命周期如何管理
- 一个进程里加载几个模型实例
- 文件输入如何落盘或流式处理
- 返回结构如何标准化

也可以认为：**`InferenceEngine` 已经是“服务内核”，只是外围服务框架还不够完整。**

---

### 4.2 `asr_core/tag_parser.py`

这个模块负责把模型原始输出中的标签信息解析成结构化字段，并清洗出可直接展示的文本。

#### 当前处理内容

- 从 `<|...|>` 结构中提取标签
- 识别语言标签：如 `zh/en/yue/ja/ko/nospeech`
- 识别情绪标签：如 `HAPPY/SAD/ANGRY/NEUTRAL/...`
- 识别事件标签：如 `Speech/BGM/Laughter/Applause/...`
- 识别 ITN 模式：`withitn/woitn`
- 删除标签
- 删除 emoji
- 规范化空白字符

#### 主要输出

- `parse_metadata(text)`：把原始标签文本解析成 `metadata`
- `strip_tags(text)`：删除标签
- `strip_emojis(text)`：删除 emoji
- `normalize_whitespace(text)`：压缩空白
- `build_structured_output(raw_text)`：生成统一结构

#### 对 API 规划的重要性

这个模块说明当前系统的返回值不只是“纯转写文本”，还包含可供前端使用的结构化语义信息：

- 语言
- 情绪
- 音频事件
- ITN 模式

因此，如果未来要做正式 API，不应只设计一个 `text` 字段，而应明确：

- `raw_text` 是否保留
- `metadata` 是否作为稳定字段长期暴露
- 情绪/事件是否属于主协议的一部分

## 5. 服务层说明

### 5.1 `service/app.py`

这是当前唯一真正接上线的 HTTP 服务入口。

#### 目前已经实现的接口

- `GET /health`
- `POST /asr/chunk`

#### 服务启动方式

`run_service.cmd` 实际执行的是：

```bat
python -m uvicorn service.app:app --host 127.0.0.1 --port 18080 --reload
```

这说明当前服务更偏向**本地开发/联调模式**，不是生产部署模式。因为：

- 绑定地址是 `127.0.0.1`
- 使用了 `--reload`
- 没有进程管理、没有生产级日志、没有配置注入

#### 生命周期行为

`FastAPI` 在启动时会实例化全局 `InferenceEngine`，也就是：

- 服务启动时加载模型一次
- 服务关闭时清理会话缓存

这对 API 改造非常关键，因为它表明当前设计已经默认“模型驻留内存，不是每个请求临时加载”。

---

### 5.2 `/health` 的现状

当前 `/health` 会返回：

- 服务是否在运行
- 模型是否加载成功
- 当前设备（CPU/GPU）
- 模型名

这个接口已经具备最基础的健康检查能力，但还比较轻量。它更像“联调可用性检查”，不是严格意义上的生产健康检查。

---

### 5.3 `/asr/chunk` 的现状

这是当前项目最重要的在线识别接口。

#### 当前请求方式

`multipart/form-data`

字段包括：

- `file`
- `session_id`
- `chunk_index`
- `is_final`
- `language`
- `use_itn`

#### 当前处理逻辑

这个接口不是“真正的流式识别”，而是一个**伪流式 / 分片重跑模式**：

1. 前端上传一个 chunk。
2. 后端把这个 chunk 追加到内存中的 `session_storage[session_id]`。
3. 后端把当前会话累计的全部音频字节写到一个临时文件。
4. 用 `InferenceEngine.transcribe_file(...)` 对“整段累计音频”重新识别一遍。
5. 返回当前累计结果。
6. 如果 `is_final=true`，清理该会话缓存。

#### 这意味着什么

这不是模型级增量推理，而是：

- 第 1 片：识别第 1 片
- 第 2 片：识别第 1+2 片
- 第 3 片：识别第 1+2+3 片

所以随着 session 增长：

- 单次请求耗时会越来越长
- 内存占用会增加
- 临时文件写入量会增加
- 并发能力会受到明显影响

这对 API 规划是一个核心约束，必须明确让规划者知道：  
**当前在线方案本质上是“把离线单文件识别包装成会话式接口”，而不是原生流式 ASR 服务。**

---

### 5.4 `service/` 中其他文件的状态

当前 `service/` 目录除了 `app.py` 外，还有：

- `config.py`
- `schemas.py`
- `session_manager.py`
- `logger.py`

这些文件目前更像是**迁移中的占位模块**，不是完整接入的正式架构。

#### `service/config.py`

定义了一个简单的 `ServiceSettings` 数据类，里面有：

- app title/version
- 默认语言
- 默认 use_itn
- 模型名
- health status

但目前 `service.app` 并没有真正使用这套配置对象，所以它还不构成完整配置体系。

#### `service/schemas.py`

定义了若干 `TypedDict`，试图把接口返回结构标准化，例如：

- `MetadataDict`
- `HealthDataDict`
- `ChunkResultDict`
- `ApiEnvelopeDict`

但当前 `app.py` 并没有统一使用这些 schema 做请求/响应约束，也没有接入 `Pydantic` response model。

#### `service/session_manager.py`

封装了一个线程安全的内存会话管理器，支持：

- `append_chunk`
- `pop_session`
- `clear`
- `has_session`
- `session_size`

但当前 `app.py` 仍然直接使用模块级全局变量 `session_storage + Lock`，没有真正用这个类。

#### `service/logger.py`

只是一个最小 logger 包装，目前也没有被服务主逻辑真正接入。

#### 结论

`service/` 已经有“分层设计”的意图，但现在仍处于“准备重构、还未完成接线”的状态。

对于 API 规划人员来说，这一点很重要：  
**当前代码不是完全杂乱，而是已经开始往更清晰的服务架构迁移，只是迁移还没完成。**

## 6. 脚本层说明

### 6.1 `scripts/smoke_test_sensevoice.py`

用途：对单个音频做冒烟测试。

特点：

- 直接加载 `AutoModel`
- 直接打印原始结果和清洗结果
- 更像“验证模型环境是否可用”的脚本

这类脚本主要用于开发初期确认：

- 环境是否安装正确
- 模型是否能正常下载/加载
- 单条音频是否能跑通

---

### 6.2 `scripts/infer_dir.py`

用途：对一个目录里的音频文件批量识别并输出 TSV。

特点：

- 遍历目录中的音频文件
- 对每个文件调用模型识别
- 输出 `utt_id<TAB>pred_text`
- 出错写到 `logs/infer_errors.log`

需要注意的是，这个脚本**没有复用 `InferenceEngine`**，而是自己重新写了一套模型加载和推理逻辑，所以它代表的是较早阶段的实现方式。

对 API 规划人员来说，这说明当前仓库里存在一定程度的逻辑重复，后续服务化过程中值得统一。

---

### 6.3 `scripts/eval_batch.py`

用途：用共享核心引擎对目录做批量识别，并生成 `pred.tsv`。

这个脚本比 `infer_dir.py` 更接近当前主路线，因为它：

- 已经复用了 `asr_core.engine.InferenceEngine`
- 把结果里的 `text` 作为预测文本写入 TSV
- 对单文件失败做日志记录但不中断整个批处理

这个脚本可以看作“离线批处理版的标准调用方式”。

---

### 6.4 `scripts/evaluate.py`

用途：读取参考 TSV 和预测 TSV，计算评测指标。

当前具备：

- 按 `utt_id` 对齐参考和预测
- 对文本做统一预处理
  - 去标点
  - 去空白
  - 小写化
- 计算：
  - `WER`
  - `SER`
- 输出到 `outputs/metrics.json`

这表明当前仓库除了识别能力，还保留了一个最基础的“可衡量效果”的评测闭环。

---

### 6.5 `scripts/temp_clean_ref_tags.py`

用途：对参考 TSV 去除 `<|...|>` 标签。

这说明评测数据与模型原始输出格式之间存在一次额外的对齐工作，进一步证明：

- 原始输出是带标签的
- 评测用文本通常是清洗后的纯文本

这个事实在未来 API 设计中也会影响字段定义：  
调用方到底要原始文本还是纯净文本，最好在协议层提前明确。

## 9. 当前“API 化”进度评估

### 9.1 已经具备的 API 化基础

当前项目已经有以下可直接复用的 API 化基础：

- 有独立的核心推理模块 `InferenceEngine`
- 模型只在服务启动时加载一次
- 已有 `FastAPI` 服务入口
- 已有健康检查接口
- 已有一个 chunk 上传接口
- 已经能把识别结果封装成结构化 JSON

这意味着项目不是从零开始做 API。

---

### 9.2 还不算“成熟 API 服务”的原因

尽管已经有服务入口，但从工程角度看，它还不能算成熟 API 服务，原因主要有：

- 接口数量少，目前只有 `health` 和 `chunk`
- 请求/响应 schema 没有统一收口
- `service/` 中的配置、schema、session manager 还未完全接线
- 会话状态全部存在进程内内存，无法跨进程共享
- chunk 模式本质上是“全量重跑”，性能会随会话时长恶化
- 依赖临时文件落盘，不是纯内存/流式处理
- 错误码、HTTP 状态码、异常返回还没有系统化
- 没有完善测试
- 没有生产级日志和配置管理

所以更准确地说，当前项目状态是：

**“已有 API 原型，但仍属于离线项目向在线服务迁移中的中间版本。”**

## 10. 当前实现与文档草案之间的差距

仓库根目录已经存在两份与接口相关的文档：

- `ASR接入.md`
- `API设计说明.md`

从代码结构和命名可以看出，这两份文档代表的是“接入说明”和“目标 API 协议草案”。  
但当前实现与文档目标之间仍存在几个明显差距。

### 10.1 `POST /asr/file` 尚未真正落地

文档草案里已经出现了单文件上传接口的设计意图，但当前 `service/app.py` 实际只实现了 `POST /asr/chunk`，还没有正式的 `POST /asr/file`。

---

### 10.2 统一响应包裹尚未彻底落实

`service/schemas.py` 里已经有 `ApiEnvelopeDict`，说明设计意图是统一成类似：

- `ok`
- `code`
- `message`
- `data`

但当前 `app.py` 实际返回仍然是“直接业务字段 + 部分错误字段”，没有彻底统一。

---

### 10.3 `empty final chunk` 协议与代码存在潜在不一致

`demo.html` 的停止录音逻辑会额外发送一个空的 `Blob`，并带上 `is_final=true`，这说明前端联调思路里存在“只发结束信号、不发音频内容”的场景。

但当前 `service/app.py` 的实现里，只要收到空文件就直接报错；它虽然会在 `finally` 中清理 session，但不会把“空 finalize”视为合法成功请求。

这说明：

- 前端 demo 的预期
- API 设计草案的方向
- 当前后端实现

三者之间还没有完全对齐。

这类差异对 API 规划尤其重要，因为它会直接影响会话协议设计。

---

## 11. 对制定 API 改造计划的人最重要的现实约束

这一节不是在替对方做方案，而是在说明“做方案前必须知道的事实”。

### 11.1 当前在线识别不是原生流式 ASR

当前 `chunk` 接口的本质是：

- 收一个新片段
- 拼回整个会话音频
- 对整个会话重新跑一次单文件识别

所以 API 规划必须先定清楚：

- 目标是不是“实时流式识别”
- 如果是，是否继续沿用当前方案
- 如果不是，是否只做“短会话伪实时识别”

这是最核心的架构问题之一。

---

### 11.2 当前会话状态是进程内内存态

会话缓存放在 `service.app` 的全局字典中，特点是：

- 服务重启即丢失
- 不能天然横向扩容
- 多 worker/多实例时无法共享
- 只适合单进程联调

因此，是否要保留内存态、换 Redis、换文件缓存、还是彻底取消 session 化，属于 API 规划必须面对的问题。

---

### 11.3 当前推理输入依赖本地临时文件

`/asr/chunk` 并不是把二进制直接喂给模型，而是先把累计 bytes 写成临时文件，再把路径交给 `InferenceEngine`。

这意味着规划时要考虑：

- 临时目录管理
- 文件清理失败时的风险
- 磁盘 I/O 开销
- 高并发下的文件系统压力

---

### 11.4 当前项目的“配置中心”还没有成型

虽然有 `service/config.py`，但实际运行参数仍然大多写死在代码中，例如：

- 模型名
- 默认语言
- 端口
- host
- 是否 reload
- CORS 策略

因此规划 API 化时，配置系统通常会是一个独立工作项。

---

### 11.5 当前测试基础比较薄弱

`test/` 目录目前基本为空，说明：

- 自动化测试体系尚未建立
- 接口契约、异常场景、并发场景缺少测试保护

如果另一个人要制定 API 改造计划，通常要把“先补测试还是边改边补测试”纳入计划。

---

## 12. 站在 API 规划视角看，这个仓库最像什么

最准确的描述是：

**一个已经完成 ASR 核心能力验证，并开始做 FastAPI 服务封装，但尚未完成工程化收口的项目。**

它不是：

- 纯研究代码
- 纯 demo
- 完整生产服务

它处在中间态：

- 核心识别能力已经能用
- 接口原型已经能跑
- 工程边界已经出现
- 但服务层治理还没有完成

这个定位对制定计划很重要，因为它意味着后续工作不应该从零推翻，而更适合：

- 保留 `asr_core`
- 整理 `service`
- 统一脚本与接口的调用方式
- 补齐配置、协议、错误处理、测试、部署能力

## 13. 建议交给规划者优先阅读的文件顺序

如果对方之后需要少量看代码，而不是通读全部仓库，推荐阅读顺序如下：

1. `asr_core/engine.py`
   - 先理解真正的推理核心。
2. `asr_core/tag_parser.py`
   - 再理解为什么返回值里有 `metadata`。
3. `service/app.py`
   - 再看当前 HTTP 服务到底做了什么。
4. `service/schemas.py`
   - 看项目已经有哪些响应标准化意图。
5. `service/session_manager.py`
   - 看当前会话管理未来准备如何抽离。
6. `scripts/eval_batch.py`
   - 看离线批量调用是如何复用核心引擎的。
7. `scripts/evaluate.py`
   - 看当前项目如何定义“效果评估”。
8. `demo.html`
   - 看前端联调时对 chunk 接口的实际预期。

## 14. 可直接转交他人的总结版

如果需要把这份说明压缩成几句话给另一个人，可以直接转述为：

1. 这个仓库的核心能力已经沉淀在 `asr_core/engine.py`，本质上是一个基于 SenseVoiceSmall 的本地文件识别引擎。
2. 当前 API 层在 `service/app.py`，已经有 `FastAPI` 原型，支持 `GET /health` 和 `POST /asr/chunk`，但还没有形成完整、标准化、可扩展的服务架构。
3. 当前 `chunk` 接口不是原生流式识别，而是把同一 `session` 的所有音频在内存中累计后，每次重新识别整段音频，所以性能和并发能力会随会话变长而变差。
4. `service/config.py`、`schemas.py`、`session_manager.py`、`logger.py` 表明项目已经开始做服务层分层，但这些模块还没有完全接入主流程。
5. `scripts/` 目录里保留了批量识别和评测能力，说明当前项目更像“离线 ASR 项目向在线 API 项目迁移中的版本”，适合在保留核心引擎的前提下做服务化规划，而不是整体推翻重写。
